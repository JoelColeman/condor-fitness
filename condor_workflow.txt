CONDOR FITNESS
Workflow Reference

How Chat, Code, and Joel work together to build the
workout companion and dashboard.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
THE THREE ROLES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CHAT (Claude in claude.ai)
  Strategic advisor and project manager. Handles program
  design, exercise selection, weight targets, progression
  logic, and what to build next. Owns athlete.json and
  program.json — these files are never edited by Code.
  Produces prompts for Claude Code. Reads build.md and
  index.html after Joel syncs the repo to the project.

JOEL
  The bridge. Carries decisions and prompts from Chat to
  Code, and carries build results back. Approves plans
  before execution. Merges PRs. Commits athlete.json and
  program.json to the repo. Syncs project files so Chat
  can read the latest repo state.

CODE (Claude Code in terminal)
  Builder and executor. Reads condor-build.md for task
  context. Commits to a branch and provides a PR merge
  link. Updates condor-build.md to reflect completed work.
  Never pushes directly to main. Never edits athlete.json
  or program.json.

INFORMATION FLOWS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CHAT → Joel → CODE
  Program decisions, build specs, prompt drafts for
  Claude Code.

CODE → condor-build.md
  After each build task, Code updates condor-build.md
  with completed status and any new tasks or issues.

Repo → Chat (via sync)
  Joel syncs the GitHub repo to this Claude project so
  Chat can read the live build doc, index.html, and
  session data.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DOCUMENT OWNERSHIP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

athlete.json
  Owned by Chat. Lives in the repo. Contains the full
  athletic history, PRs, timeline blocks, body weight
  history, goals, and skill status. Joel commits updated
  versions when Chat produces them. Code never edits this.

program.json
  Owned by Chat. Lives in the repo. Contains the full
  active training program — all weeks, days, exercises,
  weights, and alternate workouts. Joel commits updated
  versions when the program changes. Code never edits this.

condor-build.md
  Owned by Claude Code. Lives in the repo. Contains build
  state, completed tasks, technical conventions, and known
  issues. Code updates this as part of every commit.
  Joel syncs it to the project so Chat can read it.

CLAUDE.md (optional)
  Instructions for Claude Code agents. Owned by Chat/Joel.
  Tells Code how to behave, what conventions to follow,
  and how to update the build doc on task completion.

index.html
  The live workout companion app. Owned by Claude Code.
  Single-file HTML/CSS/JS. Never edited manually.

sessions/ folder
  Session JSON files written by the companion app via the
  GitHub API. One file per completed workout. Never edited
  manually. Code sets up the folder structure; the app
  writes the records.

CLAUDE CODE PROMPT CONVENTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every prompt handed to Claude Code must follow this structure:

- Start with a planning step — describe what will change
  and ask Code to confirm before executing.

- End with: "After committing, provide the direct PR merge
  link to merge your branch into main so I can merge with
  one click."

- Include: "If you find yourself retrying the same approach
  more than twice, stop and tell me what's blocking rather
  than continuing to loop."

- Include an instruction to update condor-build.md as part
  of the commit — marking tasks complete and noting
  anything new.

TECHNICAL CONVENTIONS

- Large file writes: always use bash heredoc
  (cat > index.html << 'PYEOF'). Never use the Write tool
  for files over ~400 lines — it silently drops content.

- After any commit: verify the git diff matches expected
  changes before moving to the next task.

- Start a fresh agent for each distinct build phase.

- Claude Code cannot push directly to main — manual PR
  merge by Joel is always required.

- athlete.json and program.json are read-only for Code.
  Never modify, overwrite, or regenerate these files.
  If a build task requires data from them, read them.

- session files in sessions/ are written by the app
  via GitHub API, not by Code directly.

PROJECT FILE SYNC WORKFLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The GitHub repo is connected to this Claude project.
Syncing makes the latest repo files readable by Chat.

- After Claude Code merges a PR: Joel re-syncs the project
  files in claude.ai settings.

- The sync has a lag — allow a minute or two before
  expecting Chat to see updated content.

- athlete.json and program.json live in the repo and are
  readable by Chat after sync.

- The full repo is synced: index.html, condor-build.md,
  sessions/ folder, and any other committed files are all
  readable by Chat after sync.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
AGENT HYGIENE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Claude Code agents accumulate context debt. Keep them clean.

- Start a fresh agent for each distinct build phase or
  task group.

- If layout or CSS prompts loop more than twice without
  resolution, stop — re-spec the task in Chat before
  continuing.

- Never ask a single agent to handle both structural HTML
  changes and fine CSS polish in the same session.

- After a merge, the agent that built the PR is done.
  Start fresh for the next task.

TWO-BUILD PLAN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUILD 1 — Workout Companion (index.html)
  Phone-first web app. Reads program.json to render
  workouts sequentially. User checks off sets, logs
  actual weights and times, rates calf and skills,
  saves session JSON to sessions/ via GitHub API.
  No API keys. No frameworks. Single file.
  Status: Build prompt ready. Pending Code execution.

BUILD 2 — Training Dashboard (dashboard.html)
  Phone and desktop. Reads athlete.json and sessions/.
  Displays full athletic timeline, PR board, body weight
  trend, calf/shoulder/grip skill history, run progression,
  phase checkpoints, and forward-looking goals.
  Status: Planned. Build after real session data exists.

THE BRIDGE
  Both builds share the same JSON schema. Build 1 writes
  it, Build 2 reads it. When program changes, Chat produces
  an updated program.json and Joel commits it — the app
  updates automatically without a rebuild.

SITUATION                          ACTION
─────────────────────────────────────────────────────
Program changes (new week,         Chat produces updated program.json.
weights, exercises)                Joel commits to repo. App updates
                                   automatically.

Athlete history needs updating     Discuss in Chat. Chat produces
(new PR, weight, milestone)        updated athlete.json. Joel commits.

Ready to build something           Get a prompt from Chat. Hand to
                                   Claude Code. Confirm plan before
                                   executing.

Build task complete, PR merged     Re-sync project files. Start fresh
                                   agent for next task.

Something breaks in a build loop   Stop the agent. Bring problem back
                                   to Chat. Re-spec before retrying.

Session data looks wrong in app    Check sessions/ JSON files in repo.
                                   Report to Chat — may need schema fix.

New training block starts          Chat produces new program.json for
                                   the block. Joel commits. Pointer in
                                   localStorage resets or continues.

Want to see progress in dashboard  Sync repo to project. Chat reads
                                   session files and athlete.json.
                                   Reports on progress or updates files.
